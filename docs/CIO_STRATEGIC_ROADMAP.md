## Executive Summary

After reviewing both the architectural identity audit and the functional ITSM assessment, the overarching conclusion is clear: **The system is trying to be both an Identity Provider and an IT Helpdesk, but it currently lacks the foundational safety rails for both.**

However, the system's monolithic design is a strength for an LGU with limited ICT resources. The goal is not to split the system into microservices, but to harden its internal boundaries and build out missing capabilities incrementally.

**The Verdict:** The immediate priority is not to build a complex ecosystem Identity API, nor is it to build a complex Notification Hub. The immediate priority is to **stop live data leaks, establish accountability, and make the helpdesk actually function as a service desk.**

---

## 1. Critical Blockers (Fix Immediately)

These are live bugs, data vulnerabilities, and compliance failures that transcend any roadmap phase. They must be patched immediately.

| Blocker | Domain | Impact | Why it's critical |
|:---|:---:|:---:|:---|
| **Ticket Scoping by Name String** | Security / ITSM | Critical | Employees see others' tickets if they share a name; name changes orphan tickets. This is a live data authorization leak. |
| **No Ticket Assignment** | ITSM | Critical | ICTO cannot manage workload; zero accountability for ticket resolution. |
| **Missing Fallback Secrets** | Security | Critical | JWT and OAuth fallbacks (`'fallback_secret'`) create silent vulnerabilities if env vars fail to load. |
| **Zero Audit Logging** | Identity | Critical | COA compliance failure; cannot prove who approved accounts, changed roles, or reset passwords. |
| **Zero Ticket Notifications** | ITSM | High | Users are blind to ticket progress unless they manually refresh the dashboard. |

## 2. Separation of Work Streams

To organize the development effort, we must separate the monolith's logical domains.

### A. Helpdesk Functionality (ITSM)
**Goal:** Transform the ticket logger into a measurable IT service management tool.
- Ticket assignment (`assignedTo`)
- SLA tracking (Response/Resolution targets)
- Status transition rules and resolution notes
- Category-based analytics and performance reporting
- Internal (private) notes for technicians

### B. Identity Foundation (IAM)
**Goal:** Ensure secure access and standardize the employee identifier across the ecosystem.
- `requireRole()` middleware (replacing inline string checks)
- Audit logging for all identity events
- Standardizing `employeeId` and migrating Google OAuth IDs
- Department management (synced from GSO)
- JWKS endpoint for secure cross-system SSO validation

### C. Notification Hub
**Goal:** Provide reliable cross-system communication.
- Currently: Only an API proxy for Brevo emails used for registration.
- Needs: Integration into ticket events (ITSM) first. Queueing and SMS can wait until volume demands it.

---

## 3. What Should Be Built First?

The development team should execute a **"Secure & Stabilize" sprint** that fixes the critical blockers across both Identity and ITSM, rather than building out one massive feature set.

If you build the Identity Core first, the helpdesk remains unusable for actual service management.
If you build ITSM enhancements first, the identity vulnerabilities remain exposed to the ecosystem.

**The First Step:** A hybrid patch phase focusing entirely on the 5 Critical Blockers identified above.

---

## 4. Next Milestone Recommendation

### Recommendation: **Option A (ITSM Maturity Enhancement) combined with Security Hotfixes**

The LGU ecosystem already relies on this system for support. The Employee Portal and GSO systems are consuming its identity adequately for now. The biggest operational gap holding the ICTO back on a daily basis is the **dysfunctional helpdesk workflow**.

You cannot manage an expanding digital ecosystem if you cannot manage the support tickets generated by it.

Therefore, the next formal milestone is **ITSM Maturity Enhancement**, but it must begin with the **Identity Security Hotfixes**.

## 5. Prioritized Roadmap

Optimized for: Small LGU ICT team, incremental delivery, high operational impact.

### Phase 1: Immediate — Secure & Stabilize (0–1 Month)
*Focus: Stop the bleeding, establish accountability, secure the perimeter.*

1. **Fix Ticket Scoping (ITSM/Sec):** Add `requesterId`, migrate data, filter by ID instead of name.
2. **Remove Fallback Secrets (Sec):** Throw fatal errors on startup if `JWT_SECRET` or OAuth keys are missing.
3. **Implement Ticket Assignment (ITSM):** Add `assignedTo` field and "My Tickets" queue for ICTO.
4. **Implement Basic Audit Logging (IAM):** Simple `AuditLog` collection for login, approval, and role changes.
5. **Implement Basic Ticket Notifications (ITSM):** Email users on ticket creation, status change, and ICTO comments.
6. **Add Password Validation (Sec):** Enforce complexity rules on the registration form.

### Phase 2: Short-Term — ITSM Core & Identity Primitives (1–3 Months)
*Focus: Measure helpdesk performance and formalize ecosystem identity rules.*

1. **Standardize `employeeId` (IAM):** Make it required, add to JWT payload, migrate `google-*` accounts.
2. **Implement `requireRole()` Middleware (IAM):** Replace fragile inline checks. Add `systemAccess` array to users.
3. **Implement SLA Tracking (ITSM):** Timestamps (`firstResponseAt`, `resolvedAt`) and basic target monitoring.
4. **Build Department Collection (IAM):** Seed from GSO, replace free-text `office` strings with references.
5. **Resolution Capture & Transition Rules (ITSM):** Require resolution notes; restrict status regressions.
6. **Publish JWKS Endpoint (IAM):** Allow PPEMS/Zoning to securely validate SSO tokens without shared secrets.
7. **Basic Analytics Dashboard (ITSM):** Build aggregation pipelines for tickets by category and time trends.

### Phase 3: Medium-Term — Ecosystem Integration (3–6 Months)
*Focus: Prepare for HRIS, enhance self-service, formalize APIs.*

1. **Identity API Versioning (IAM):** Establish `/api/v1/identity` for Portal, PPEMS, and Zoning consumption.
2. **Per-System API Keys (IAM):** Replace the single `INTERNAL_API_KEY` with specific keys for GSO, Portal, etc.
3. **Knowledge Base Foundation (ITSM):** Basic articles to deflect common tickets.
4. **HRIS Sync Preparation (IAM):** Define webhook/API contract for future HR master data ingestion.
5. **Asset Field Integration (ITSM):** Add `assetTag` to tickets in preparation for PPEMS integration.
6. **Token Introspection Endpoint (IAM):** Allow consuming systems to validate active sessions.

## 6. Overengineering to Avoid (Postpone Indefinitely)

Given the LGU context (limited developers, 500-2,000 users, single ICT office), the following recommendations from previous audits should be explicitly rejected or postponed:

| Feature | Why to Avoid / Postpone |
|:---|:---|
| **Database Separation** | Splitting `identity-db` and `helpdesk-db` adds enormous connection, backup, and querying complexity. Collection-level separation is sufficient. |
| **Custom OAuth 2.0 / OIDC Server** | Building a custom authorization server is massive overhead. Stick to the RS256 JWT redirect pattern, or adopt Keycloak if OIDC becomes legally mandated. |
| **SCIM 2.0 & SAML 2.0** | Enterprise federation protocols with zero applicability to a self-contained LGU ecosystem. |
| **BullMQ + Redis Notification Queue** | The Brevo API is reliable enough. In-process retries are fine for now. Don't add Redis infrastructure until SMS/in-app channels strictly require it. |
| **Full RBAC Permission Engine** | Creating `Permissions` collections and dynamic inheritance for 7 actual authorization rules is enterprise theater. The `requireRole()` middleware suffices. |
| **Problem & Change Management** | ITIL heavyweights. ICTO needs to master Incident Management (Phase 1/2) before attempting Problem/Change management. |

---

## Final Recommendation to the Development Team

**Stop feature development on the Employee Portal and GSO integrations momentarily.**

Begin the **Phase 1: Secure & Stabilize sprint** immediately.
Your first three tickets are:
1. Add `requesterId` to tickets and fix the scoping queries.
2. Remove the `'fallback_secret'` hardcodings.
3. Add `assignedTo` on the ticket model and UI.

This approach delivers immediate operational value to the ICTO team while quietly closing the most dangerous security gaps in the ecosystem.
